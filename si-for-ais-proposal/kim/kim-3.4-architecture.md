# КИМ-3.4. Моделирование системной архитектуры ИИ-системы

| Поле | Значение |
|---|---|
| Элемент дисциплины | Модуль 3, занятие 3.4 |
| Форма занятия | Практическое, 8 а.ч. + самостоятельная работа |
| Форма контроля | Текущая (защита ЛР) |
| Компетенции и индикаторы | **LC-3.1** — уровень С; **ОПК-4.1** — уровень Б |
| Связка инструментов | Architecture-as-Code (см. [01-tools.md](../01-tools.md)) |
| Вход | SRS, разработанный в [КИМ-2.5](kim-2.5-srs.md) |

## 1. Назначение

Проверить способность спроектировать архитектуру ИИ-системы, **обосновать** архитектурные решения и увидеть последствия их изменения. Ключевое слово — обосновать: диаграмму сегодня рисует LLM за минуту, поэтому оценивается не наличие диаграммы, а связь решений с требованиями и осознание компромиссов.

**Дескриптор освоения (LC-3.1, уровень С).** По КРМ v3.0: *«Применяет различные принципы и паттерны при проектировании архитектуры систем ИИ»*. В контексте курса: студент выбирает архитектурные паттерны под конкретные требования из собственного SRS, обосновывает выбор через trade-off с явными альтернативами и показывает, какие компоненты обеспечивают жизненный цикл ML-модели, а не только её первичный инференс.

**Дескриптор (ОПК-4.1, уровень Б).** Оформляет архитектурное описание с опорой на действующие стандарты: **ISO/IEC/IEEE 42010:2022** (архитектурные представления) и **ГОСТ Р 71539-2024 / ИСО-МЭК 5338-2023** (процессы жизненного цикла системы ИИ). Годы редакций указаны намеренно — 42010 был существенно переработан в 2022, а на 15288 в России с 30.06.2025 действует **ГОСТ Р 57193-2025**; ссылка на устаревшую редакцию в архитектурном описании является дефектом оформления.

## 2. Предметная область

Система прогнозирования спроса для сети розничных магазинов — сквозной кейс дисциплины (совпадает с примером кейса LC-3.1 в КРМ v3.0 и с проектным заданием).

Исходные ограничения:
- около 200 магазинов, горизонт прогноза 14 дней, ежедневный пересчёт;
- интеграция с ERP и WMS, выгрузка прогноза в BI;
- добавление новых категорий товаров и филиалов без переработки системы;
- поддержка полного жизненного цикла модели: от сбора данных до вывода из эксплуатации.

## 3. Условия выполнения

Работа выполняется индивидуально в личном репозитории (шаблон выдаётся через GitHub Classroom). Срок — 2 недели. Архитектурное описание ведётся **как код**: диаграммы в текстовой нотации, решения — отдельными файлами, всё под версионным контролем.

Использование ИИ-инструментов **разрешено и обязательно декларируется** (см. [модель оценивания](../02-assessment.md), раздел 5). Работа без заполненной декларации к проверке не принимается.

## 4. Задание

### Часть А. Архитектурное описание (C4, уровни 1-3)

Разработать в **PlantUML** с библиотекой **C4-PlantUML** (обе проверены на 17.07.2026: PlantUML v1.2026.6, LGPL-3.0; C4-PlantUML v2.13.0, MIT, входит в stdlib — отдельная установка не требуется).

Два предупреждения, экономящих время:

- **Structurizr Lite использовать нельзя** — репозиторий архивирован 04.02.2026 и помечен как End of life. Если нужен интерактивный просмотр модели, берётся консолидированный образ в режиме `local`: `docker run -it --rm -p 8080:8080 -v <путь>:/usr/local/structurizr structurizr/structurizr local`.
- **Mermaid для C4 не подходит.** Диаграммы C4 в Mermaid имеют экспериментальный статус (синтаксис может измениться), и GitHub их нативно **не рендерит** — в отличие от обычных Mermaid-диаграмм. Для последовательностей и простых схем Mermaid удобен, для C4 — только PlantUML.

Состав описания:

1. **Контекст (C4 L1)** — система, внешние акторы, смежные системы (ERP, WMS, BI).
2. **Контейнеры (C4 L2)** — с обязательным выделением компонентов, специфичных для ИИ-системы. Именно этот пункт отделяет ИИ-архитектуру от архитектуры обычного ПО:
   - хранилище и подготовка признаков;
   - обучающий конвейер и реестр моделей (model registry);
   - сервис инференса;
   - конвейер мониторинга качества данных и дрейфа;
   - контур переобучения с триггерами;
   - контур human-in-the-loop, если он предусмотрен требованиями.
3. **Компоненты (C4 L3)** — детализация **одного** контейнера по выбору студента; выбор обосновывается (почему именно этот контейнер наиболее рискован).
4. **Диаграмма последовательности** — сценарий ежедневного пересчёта прогноза.
5. **Диаграмма потоков данных** — с указанием точек контроля качества данных.
6. **Диаграмма развёртывания.**

Каждая диаграмма сопровождается таблицей трассируемости: `элемент архитектуры → идентификаторы требований из SRS, которые он реализует`. Элемент, не связанный ни с одним требованием, либо удаляется, либо в SRS вносится запрос на изменение.

### Часть Б. Архитектурные решения (ADR)

Оформить **три** ADR по шаблону **MADR 4.0.0** (MIT / CC0-1.0). Никакой инструментальной обвязки не требуется и не рекомендуется: ADR — это markdown-файл в репозитории, а ADR-утилиты повсеместно заброшены (`adr-tools` — без коммитов с 2020). Шаблон копируется в `decisions/`, и этого достаточно.

Обязательные темы:

1. Способ доставки прогноза потребителям (синхронный вызов или пакетный расчёт).
2. Стратегия обновления модели (переобучение по расписанию, по триггеру дрейфа или гибрид).
3. Поведение системы при недоступности или низкой уверенности модели (fallback).

Каждый ADR содержит: контекст, **не менее двух реально рассмотренных альтернатив**, критерии сравнения, решение, последствия — включая негативные. ADR без негативных последствий считается незаполненным: решение без цены означает, что компромисс не найден.

### Часть В. Инспекция сгенерированной архитектуры

Выдаётся архитектурное описание той же системы, подготовленное с помощью LLM (файл `attachments/generated-architecture.md`). Провести инспекцию: найти дефекты, классифицировать, оформить как замечания с указанием нарушенного требования.

Ожидаемый объём — не менее пяти существенных дефектов. Ищутся не опечатки, а инженерные дефекты: отсутствие мониторинга дрейфа при заявленном многолетнем сроке эксплуатации, инференс в критическом пути без таймаута, отсутствие реестра моделей при заявленном переобучении, невозможность отката к предыдущей версии модели, отсутствие точек контроля качества входных данных.

Для каждого дефекта: в чём он, какое требование SRS нарушено, каковы последствия в эксплуатации, как исправить.

### Часть Г. Анализ влияния изменения

Вводная выдаётся **на защите**, готовиться к ней заранее нельзя:

> Заказчик уточнил: данные не могут покидать контур предприятия, развёртывание только on-premise, доступных GPU нет.

Устно, с опорой на свою архитектуру: какие контейнеры затронуты, какие ADR аннулированы, какие требования SRS становятся невыполнимыми, что предлагается взамен.

## 6. Критерии и шкала

Максимум — 100 баллов. Работа, не прошедшая контур 1, не оценивается.

| № | Критерий | Что именно проверяется | Баллы |
|:--:|---|---|:--:|
| 0 | **Допуск** (gate) | CI зелёный, все файлы на месте, декларация ИИ заполнена | 0 / не принято |
| 1 | Полнота описания | Шесть диаграмм, ИИ-специфичные контейнеры выделены | 10 |
| 2 | Трассируемость | Каждый элемент связан с требованием; «сирот» нет | 15 |
| 3 | Качество ADR | Реальные альтернативы, явные критерии, названы негативные последствия | 15 |
| 4 | Инспекция (часть В) | Найденные дефекты, корректность классификации, связь с требованиями | 15 |
| 5 | Защита содержания | Отвечает «почему так», видит последствия решений | 25 |
| 6 | Анализ влияния (часть Г) | Определяет затронутые элементы, аннулированные ADR, предлагает альтернативу | 10 |
| 7 | Контроль над ИИ | Качество декларации, честность в описании непроверенного | 10 |

**Шкала:** 85-100 — отлично; 70-84 — хорошо; 55-69 — удовлетворительно; < 55 — не зачтено.

**Отдельное правило.** Работа, набравшая максимум по критериям 1-4, но менее 10 из 25 по критерию 5 (защита), считается несданной независимо от суммы. Это прямое следствие принципа «артефакт без защиты не оценивается»: документ, который студент не может объяснить, не является результатом его работы.

### Пояснение к критерию 3 (для проверяющего)

Типичная ошибка — ADR, где альтернативы приведены формально, а «выбранный» вариант очевиден с самого начала. Признак: негативные последствия отсутствуют или сформулированы как «требует чуть больше ресурсов». Признак хорошего ADR — студент называет цену своего решения так, что видно: он понимает, чем платит.

Пример разницы на теме fallback:

- **Формально:** «Выбран fallback на скользящее среднее, так как это надёжно. Минусы: точность ниже».
- **Содержательно:** «Выбран fallback на скользящее среднее по последним 28 дням. Альтернатива — отдавать последний успешный прогноз — отвергнута: при отказе конвейера в предпраздничную неделю она законсервирует пиковые значения и WMS сформирует избыточный заказ. Цена решения: в момент срабатывания fallback система теряет учёт акций, ошибка по промо-позициям вырастет ориентировочно вдвое; поэтому требуется алерт менеджеру категории — REQ-031».

## 7. Правила использования ИИ и внешних ресурсов

Разрешено: LLM для черновиков диаграмм, ADR и текста; поисковые ИИ-сервисы для аналогов; любые открытые источники.

Обязательно:
- декларация по форме из [модели оценивания](../02-assessment.md);
- **осмысленная история коммитов.** Единственный коммит со всей работой не является нарушением, но служит основанием для расширенного опроса на защите;
- фактическая проверка: если LLM ссылается на паттерн, стандарт или цифру — студент открывает первоисточник. Ссылка на несуществующий стандарт снимает баллы по критерию 7 полностью.

Не разрешено: сдавать сгенерированный текст без проверки. Практическая граница проста — на защите задаётся вопрос по любому фрагменту.
